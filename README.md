# Bookmark Advisor

An AI-first Microsoft Edge bookmark organizer.

The system now has three main parts:

- Python exports bookmark snapshots and generates AI draft plans
- rules in `config/rules.yaml` act as guardrails and forced fallbacks
- the unpacked Edge extension applies reviewed plans with `chrome.bookmarks.*`
- skills in `skills/` teach coding agents how to review public URLs and orchestrate the full reorg flow

## Recommended workflow

1. Export a snapshot of the current Edge bookmarks.
2. Build a review queue for public URLs.
3. Complete URL review and merge it into an enriched snapshot.
4. Ask the AI planner to create a draft semantic reorganization plan.
5. Finalize the draft into a reviewed plan.
6. Generate a reviewed plan directly in the Edge extension via HTTPS, or import a reviewed plan generated by the CLI.
7. Optionally export a fresh snapshot and diff it against the previous one.

## Core commands

Validate rules:

```bash
PYTHONPATH=src python3 -m bookmark_advisor validate-rules --rules config/rules.yaml
```

Export a snapshot:

```bash
PYTHONPATH=src python3 -m bookmark_advisor export-snapshot
```

Initialize an end-to-end reorg job:

```bash
PYTHONPATH=src python3 -m bookmark_advisor init-job \
  --workspace . \
  --primary-backend extension
```

Build a review queue from a snapshot:

```bash
PYTHONPATH=src python3 -m bookmark_advisor build-review-queue \
  --snapshot data/snapshots/snapshot_YYYYMMDD_HHMMSS.json
```

Merge snapshot and reviewed URL sidecar:

```bash
PYTHONPATH=src python3 -m bookmark_advisor enrich-snapshot \
  --snapshot data/snapshots/snapshot_YYYYMMDD_HHMMSS.json \
  --reviews data/reviews/url_review_YYYYMMDD_HHMMSS.json
```

Generate an AI draft plan:

```bash
OPENAI_API_KEY=... PYTHONPATH=src python3 -m bookmark_advisor plan-ai \
  --snapshot data/snapshots/enriched_snapshot_YYYYMMDD_HHMMSS.json \
  --rules config/rules.yaml \
  --model gpt-4o-mini
```

Generate an AI draft plan through an OpenAI-compatible endpoint:

```bash
OPENAI_API_KEY=... OPENAI_BASE_URL=https://your-provider.example/v1 \
PYTHONPATH=src python3 -m bookmark_advisor plan-ai \
  --snapshot data/snapshots/snapshot_YYYYMMDD_HHMMSS.json \
  --rules config/rules.yaml \
  --model your-provider-model \
  --api-style auto
```

Finalize a draft plan into a reviewed plan:

```bash
PYTHONPATH=src python3 -m bookmark_advisor finalize-plan \
  --input data/plans/draft_YYYYMMDD_HHMMSS.json
```

Compare two snapshots:

```bash
PYTHONPATH=src python3 -m bookmark_advisor diff-snapshot \
  --before data/snapshots/snapshot_before.json \
  --after data/snapshots/snapshot_after.json
```

Run the next available phase in a `reorg-job.json` workflow:

```bash
PYTHONPATH=src python3 -m bookmark_advisor run-job \
  --job data/jobs/reorg_YYYYMMDD_HHMMSS/reorg-job.json
```

Legacy heuristic planning still exists:

```bash
PYTHONPATH=src python3 -m bookmark_advisor merge
PYTHONPATH=src python3 -m bookmark_advisor advise
```

## Open files, not hidden state

The main intermediate files are:

- `snapshot.json`: exported bookmark state
- `review-queue.json`: public bookmark URLs that require semantic review
- `url-review.json`: independent URL review sidecar
- `enriched-snapshot.json`: snapshot merged with URL review evidence
- `draft-plan.json`: AI-generated proposal
- `reviewed-plan.json`: approved plan for extension execution
- `execution-report.json`: extension execution result
- `reorg-job.json`: workflow manifest for coding agents and CLI orchestration

These are meant to be editable and reusable. The extension is an executor, not the only place where logic lives.

## Rules

Rules are stored in [config/rules.yaml](/Users/diekunstderfuge/myrepos/edge-bookmark/config/rules.yaml:1).

Rules are intentionally limited to guardrails:

- protect specific roots or folders
- force a few known relocations
- constrain risky actions
- provide fallback behavior when the AI is uncertain
- keep loose bookmarks under protected roots in place by default

Protected-root default:

- If `protect_root_loose_bookmarks: true`, loose bookmarks directly under `/收藏夹栏` or any other `protected_paths` root are not auto-reorganized.
- Those root-level loose bookmarks may remain in place or become `keep_for_review`.
- They should only be reorganized after an explicit user request for root-level cleanup.

## OpenAI SDK compatibility

The AI planner now uses the official Python `openai` SDK instead of a hand-written HTTP client.

It supports:

- OpenAI directly with `OPENAI_API_KEY`
- OpenAI-compatible providers with `OPENAI_BASE_URL`
- automatic fallback from `responses` to `chat.completions`
- an additional fallback to `json_object` mode for providers that do not support strict JSON schema output

Useful knobs:

- `OPENAI_API_KEY`: provider API key
- `OPENAI_BASE_URL`: root API base such as `https://api.openai.com/v1` or a compatible provider root
- `OPENAI_API_STYLE`: `auto`, `responses`, or `chat_completions`
- `--base-url`: CLI override for `OPENAI_BASE_URL`
- `--api-style`: CLI override for `OPENAI_API_STYLE`

For backward compatibility, if `OPENAI_BASE_URL` is accidentally set to a full endpoint like `.../v1/responses`, the planner will normalize it back to the provider root before creating the SDK client.

## Edge extension

The unpacked extension lives in [extension](/Users/diekunstderfuge/myrepos/edge-bookmark/extension:1).

Use it like this:

1. Open `edge://extensions`
2. Enable developer mode
3. Load the `extension/` directory as an unpacked extension
4. Open the `LLM Settings` tab and configure an HTTPS OpenAI-compatible base URL, model, endpoint mode, and API key
5. Return to the `Plan` tab and click `Generate AI Plan`, or import a reviewed plan JSON file
6. Review the preflight summary
7. Execute the reviewed plan inside Edge
8. Optionally export the current snapshot from the extension
9. Optionally download the execution report JSON

The HTTPS planner path is intentionally SDK-free: the extension uses `fetch` against OpenAI-compatible REST APIs. In `auto` mode it tries Responses API JSON schema first, then Chat Completions JSON schema, then Chat Completions JSON object/plain JSON fallbacks for compatible providers that do not support Structured Outputs.

The popup can save provider credentials locally. The API key is stored in `chrome.storage.local` as AES-GCM ciphertext, with the AES key derived automatically from the extension install using SHA-256. This avoids a separate unlock password, but it is convenience encryption rather than protection from someone who already controls the browser profile or extension storage.

The popup also auto-saves in-progress form drafts because browser extension popups are destroyed when they lose focus. Provider URL, model, focus folder, max actions, selected tab, and the current API key draft are restored when the popup is reopened. API key drafts use the same AES-GCM storage path as saved keys.

Fast extension planning uses bookmark titles, URLs, domains, and folder paths as evidence. For high-stakes semantic cleanup, the fuller CLI flow with URL review sidecars remains the stricter path.

The popup now performs two validation layers before execution:

- JSON syntax validation in-browser when the file is loaded
- plan-shape linting for required fields, unsupported action types, and missing locator data

If you only want a quick syntax check from Python, the standard library already has one:

```bash
python3 -m json.tool data/plans/reviewed_plan.json
```

That built-in check is useful for raw JSON syntax only. It does not know the Bookmark Advisor plan shape, so the extension linter is still the source of truth for whether a reviewed plan is executable.

## Fallback mode

`--apply --write-source` is still available as a debugging or recovery path, but it is not the recommended workflow because direct file edits can be overwritten by Edge sync.
